# Meem 转向浏览器插件形态

日期：2026-05-21

## 决定

Meem 从当前 Codex / 桌面客户端形态，转向 `meem-extension`。浏览器插件成为产品主体，仓库根目录直接采用插件版代码。

这次调整不做兼容层，不保留旧 Codex bridge、桌面客户端、旧主 Worker 和旧浏览器控制实验作为主线。旧代码只进入备份，不继续作为开发入口。

## 原因

Meem 的核心是接收外部信息、理解上下文、生成回应，并在用户的浏览器环境里辅助处理任务。插件形态更贴近这个场景：

- 浏览器是用户接收和处理信息的主要现场。
- 插件可以直接获得网页、标签页、侧栏和当前上下文。
- 用户不需要先理解桌面客户端、Codex bridge、本机服务等开发形态。
- 产品入口更轻，安装后即可围绕网页和收件开始使用。

## 新边界

- `src/`：插件侧栏、AI 处理、浏览器工具和本地配置。
- `public/manifest.json`：Chrome Extension 配置。
- `worker/`：插件自己的 Cloudflare Worker，用于分身、收件和外部消息接口。
- `dev/doc/`：产品和工程决策文档。
- `dev/bak/`：重构前代码快照。

旧的 `client/`、`server/`、根部 `worker/`、`browser-use/` 不再是主产品代码。

## 执行方式

1. 在 `dev/doc/` 记录本次产品决策。
2. 在 `dev/bak/` 保存当前代码压缩包。
3. 将 `extension/` 内容全量提升到仓库根目录。
4. 删除旧主线目录，只保留插件产品所需结构。

## 后续原则

- 先做插件完整闭环，再扩展后台能力。
- 不恢复 Codex 依赖。
- 不把旧架构以兼容方式塞回新代码。
- 命名以插件产品为准，避免同时存在两套 Meem 概念。
